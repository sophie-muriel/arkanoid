1. Introducción

La especificación de Requisitos de Software (ERS) para Arkanoid Evolution (ArkEv) establece de manera estructurada los requisitos funcionales y no funcionales que guiarán el desarrollo del videojuego arcade, cuyo objetivo es ofrecer una experiencia modernizada del clásico Arkanoid.

1.1 Propósito

El propósito de este documento es especificar de forma clara y detallada los requisitos funcionales y no funcionales del videojuego Arkanoid Evolution (ArkEv). Está diseñado para guiar a los equipos de desarrollo, diseño, pruebas, así como al Product Owner (PO), Project Manager (PM) y al cliente, durante todo el ciclo de vida del software. Este documento busca establecer una base de entendimiento común que garantice el desarrollo correcto del videojuego, cumpliendo las expectativas de calidad, experiencia de usuario y rendimiento.

1.2 Ámbito del sistema

Arkanoid Evolution (ArkEv) es un videojuego inspirado en el clásico Arkanoid, en el que el usuario controla una plataforma para hacer rebotar una bola con el objetivo de romper bloques dispuestos en distintos patrones. El sistema incluirá una estructura de niveles progresiva, un sistema de puntuación, variedad de power ups, efectos visuales modernos, sonidos envolventes, y una banda sonora retro. ArkEv será una aplicación de escritorio ejecutable localmente, sin modos de juego multijugador ni conectividad permanente, pero incluirá una funcionalidad en línea opcional para el registro y consulta de rankings y puntuaciones. Este sistema de ranking permitirá a los jugadores comparar su progreso y logros con los de otros jugadores a través de una conexión puntual con el servidor de rankings.

Se espera que ArkEv ofrezca una experiencia nostálgica pero renovada, atrayendo tanto a jugadores veteranos como a nuevas generaciones, gracias a una jugabilidad sencilla, mecánicas clásicas y un entorno audiovisual atractivo.

1.3 Definiciones, acrónimos y abreviaturas

  • Arkanoid: Estilo de juego en el que el jugador destruye bloques con una pelota que rebota.
  • PM: Project Manager.
  • PO: Product Owner.
  • Power Up: Elemento especial que otorga habilidades o ventajas temporales al jugador.
  • FPS: Frames Per Second (fotogramas por segundo).
  • IA: Inteligencia Artificial.
  • Ranking en línea: Sistema remoto que permite registrar y consultar puntuaciones de jugadores.
  • Skin: Apariencia visual alternativa que modifica el diseño gráfico de elementos del juego sin afectar su comportamiento ni las mecánicas del mismo.
  • ERS: Especificación de Requisitos de Software. Documento que describe en detalle las funcionalidades, restricciones y requerimientos técnicos de un sistema o aplicación, siguiendo un estándar definido como IEEE 830.
  • IEEE: Institute of Electrical and Electronics Engineers. Organización internacional que establece estándares técnicos para diversas áreas, incluyendo el desarrollo de software.
  • IEEE 830: Estándar para la especificación de requisitos.
  • Unity: Motor de desarrollo de videojuegos multiplataforma que permite crear entornos interactivos en 2D y 3D.
  • API/RESTful: Application Programming Interface / Representational State Transfer. Tipo de interfaz de programación que permite la comunicación entre aplicaciones mediante el protocolo HTTP, facilitando operaciones como el envío y recuperación de datos.

1.5 Visión general del documento

Este documento se divide en cuatro secciones principales, a saber: una introducción que describe el propósito, alcance, definiciones y estructura del documento; una descripción general del sistema, donde se presenta una visión global del producto, sus funciones, usuarios, restricciones y dependencias; un apartado de requisitos específicos, donde se detallan los requerimientos funcionales y no funcionales del sistema y finalmente una sección de conclusiones generales donde se una reflexión sobre la importancia del presente documento y su uso futuro.

2. Descripción general

2.1 Perspectiva del producto

ArkEv será una aplicación de escritorio, ejecutable para computadores personales Windows, Linux y Mac. Funcionará sin necesidad de conexión permanente a internet, utilizará un motor gráfico ligero para 2D y podrá recibir entradas desde el teclado. Adicionalmente, incluirá un módulo de conexión temporal al servidor de rankings en línea, permitiendo el envío y la visualización de puntuaciones y logros de los jugadores, sin comprometer la jugabilidad básica del juego.

2.2 Funciones del producto

Menú principal

  • Iniciar juego
  • Ver y editar preferencias visuales y de sonido
  • Ver y editar perfil
  • Consultar las puntuaciones personales
  • Acceder al módulo de ránking global (en línea)
  • Opciones de dificultad de juego
  • Tutorial o modo de ayuda
  • Sistema de power-ups
  • Salir del juego

Gestión de perfil

  • Vista y edición de perfil incluyendo nombre de usuario y selección de país

Ránking global

  • Puesto ocupado
  • Nombre del jugador
  • País

  • Sincronización con base de datos si hay conexión
  • Mensaje de error si no hay conexión
  • Filtrado de puntajes:
    • Top 100 mayores puntajes
    • Top 100 menores puntajes
    • Puntaje propio con 100 registros en contexto

Selección de nivel

  • Continuar con el último nivel desbloqueado

Pantallas de juego

  • Opciones de dificultad: fácil, medio, difícil

Características de juego

  • Control de la plataforma con ← y → (personalizable)
  • Rebote de la bola según leyes físicas
  • Eliminación de bloques al ser golpeados
  • Sistema de power-ups:
    • Nombre

  • Sistema de puntuación:
    • Puntos por bloques destruidos

    • Disposición y resistencia de bloques

Características adicionales

  • Pantallas de victoria y derrota (por vidas o tiempo)
  • Transición entre niveles
  • Almacenamiento local de progreso y puntuaciones
  • Funcionalidad en línea opcional para rankings
  • Efectos sonoros en interacciones
  • Banda sonora para cada pantalla

2.3 Características de los usuarios

El público objetivo está compuesto por jugadores casuales y/o nostálgicos que desean disfrutar de una experiencia de juego clásica pero renovada, y que tienen a su disposición computadores sin grandes capacidades de procesamiento gráfico. Se espera que puedan usar el teclado. No se requieren habilidades técnicas avanzadas.

2.4 Restricciones

  • ArkEv será desarrollado utilizando el motor Unity, ya que este entorno cubre de forma integral las necesidades técnicas y funcionales del proyecto. Entre sus principales ventajas se encuentran el soporte nativo para gráficos 2D, físicas realistas (como colisiones y rebotes), integración de sonido y animaciones, exportación multiplataforma (Windows, Linux, Mac, WebGL), soporte para funcionalidades en línea como rankings mediante APIs, y un conjunto de herramientas visuales que facilitan la colaboración entre desarrolladores y diseñadores.
  • Todas las funcionalidades del sistema, salvo el módulo de ranking global deben poder ejecutarse completamente sin conexión a internet.
  • Solo se incluirá el modo de un jugador.
  • El módulo de ranking en línea no debe interrumpir la jugabilidad si no hay conexión disponible.
  • La funcionalidad del módulo de ranking en línea deberá ser desarrollada mediante la integración de RESTful APIs.
  • El sistema debe estar optimizado para ejecutarse correctamente en equipos de gama media o baja. A continuación se detallan las características mínimas recomendadas del equipo:
    • Procesador: Dual core a 2.0 ghz o equivalente.
    • Memoria RAM: 4 GB.
    • Almacenamiento: 4 GB de espacio libre.
    • Gráficos: gráficos integrados estándar (sin necesidad de aceleración 3D).
    • DirectX: Versión 11 o superior.
    • Tarjeta de sonido: Compatible con DirectX 11 o superior. Se acepta sonido integrado.
    • Resolución de pantalla mínima: 800x600 píxeles.
    • Sistema operativo: Windows XP o superior.
    • Conexión a internet: No requerida (solo para funciones remotas si las hubiera).
  • La disponibilidad del sistema de ranking debe estar garantizada, asegurando la integridad y seguridad de los datos almacenados.
  • La carga de los datos del jugador debe realizarse de forma rápida y eficiente.

2.5 Suposiciones y dependencias

  • Se asume que el usuario dispone de un teclado funcional para jugar.
  • Se presupone que el usuario cuenta con un computador personal que cumple como mínimo, con las especificaciones técnicas establecidas en la sección de restricciones del sistema.
  • Se requiere que el sistema operativo del usuario sea compatible con la versión del juego distribuida.
  • Se presupone que el servidor encargado de gestionar el ranking global estará disponible de forma permanente, correctamente configurado para aceptar conexiones externas y preparado para manejar de manera escalable la variabilidad en la concurrencia de usuarios.

2.6 Requisitos futuros

En futuras versiones, se espera que el sistema sea escalable a plataformas web y dispositivos móviles, con el objetivo de ampliar su alcance y accesibilidad. Asimismo, se contempla la incorporación de modos de juego en línea, como duelos en tiempo real entre jugadores, que enriquecerían la experiencia y fomentarían la competitividad. También se prevé el desarrollo de una tienda virtual, donde los usuarios puedan adquirir elementos personalizables, como skins temáticas generales, que podrían ser visualizadas y seleccionadas por los usuarios para personalizar la plataforma, bola, y fondos del juego.

3. Requisitos específicos

3.1 Interfaces externas

  • Teclado: entrada del jugador.
  • Pantalla: renderizado del juego.
  • API REST: comunicación con servidor de rankings.

3.2 Requerimientos funcionales

RF1.

El sistema debe permitir iniciar y reiniciar el juego.

RF2.

El sistema debe permitir configurar preferencias de accesibilidad y sonido, tales como modificación del tamaño de la fuente, activación o desactivación de modo de alto contraste, activación o desactivación de los efectos visuales, subir y bajar volúmen de la música, subir y bajar volúmen de los efectos de sonido.

RF3.

El sistema debe desbloquear niveles automáticamente en función del progreso del jugador.

RF4.

El sistema debe guardar localmente el progreso.

RF5.

El sistema debe permitir seleccionar el nivel a jugar, ya sea el último nivel desbloqueado o cualquier nivel previamente superado.

RF6.

El sistema debe permitir visualizar y editar la información del perfil del usuario (nombre de usuario y país).

RF7.

El sistema debe permitir al usuario abandonar el juego desde cualquier pantalla principal.

RF8.

El sistema debe permitir regresar al menú principal desde la pantalla del juego.

RF9.

El jugador debe poder mover la plataforma hacia la izquierda y la derecha usando las flechas del teclado.

RF10.

El sistema debe permitir pausar el juego al presionar la tecla Escape.

RF11.

La bola debe desplazarse y rebotar contra superficies (plataforma, paredes, bloques y obstáculos) según leyes físicas básicas.

RF12.

Los bloques deben eliminarse al ser golpeados por la bola.

RF13.

El sistema debe mostrar la animación de destrucción de bloques cuando estos sean golpeados, si los efectos visuales están activados.

RF14.

Los obstáculos deben permanecer intactos al ser chocados con la bola y deberán ser eliminados cuando sean golpeados por power-ups con la propiedad "rompe obstáculos".

RF15.

El sistema debe contar con una variedad de power-ups que otorgan habilidades o efectos temporales al jugador. La lista mínima de power-ups que el sistema debe implementar incluye:


  • Disparo: Permite al jugador disparar proyectiles para destruir bloques y obstáculos.
  • Disparo doble: Permite disparar dos proyectiles en paralelo para destruir bloques y obstáculos.
  • Disparo triple: Permite disparar tres proyectiles en abanico para destruir bloques y obstáculos.
  • +2 segundos: Agrega dos segundos al tiempo restante del nivel.
  • +5 segundos: Agrega cinco segundos al tiempo restante.
  • +10 segundos: Agrega diez segundos al tiempo restante.
  • Bola destructora vertical: Permite destruir dos bloques consecutivos en la misma columna.
  • Bola destructora horizontal: Permite destruir tres bloques consecutivos en la misma fila (el golpeado, su adyacente a la izquierda y su adyacente a la derecha).
  • Multibola: Divide la bola actual en tres, aumentando las posibilidades de impacto. Solo funciona a partir del siguiente rebote de la bola en la plataforma. Las dos bolas adicionales son de color diferente y se destruyen al chocar con un obstáculo, una pared o al pasar el límite inferior de la pantalla (sin afectar las vidas del jugador).
  • Bola reforzada: Permite a la bola atravesar hasta 3 bloques consecutivos en vertical sin rebotar.
  • Congelación: Detiene el movimiento de los bloques móviles cuando los haya.
  • Plataforma expansiva: Aumenta el tamaño de la plataforma del jugador.
  • Atracción magnética: Hace que la bola se dirija más fácilmente hacia la plataforma del jugador.
  • Ladrillo bomba: El siguiente bloque impactado por la bola destruye todos los bloques u obstáculos inmediatamente adyacentes, en área (efecto de explosión).
  • Escudo inferior: Agrega una barrera en la parte inferior que evita perder la bola (evita perder una vida) durante el tiempo que esté activo.
  • Puntos dobles: Duplica la puntuación obtenida.
  • Puntos triples: Triplica la puntuación obtenida.
RF16.

El sistema debe mantener activo el efecto de cada power-up durante un tiempo determinado, según la siguiente información:


  • Disparo: 10 segundos.
  • Disparo doble: 10 segundos.
  • Disparo triple: 10 segundos.
  • +2 segundos: Efecto inmediato.
  • +5 segundos: Efecto inmediato.
  • +10 segundos: Efecto inmediato.
  • Bola destructora vertical: Un impacto.
  • Bola destructora horizontal: Un impacto.
  • Multibola: Hasta que las bolas adicionales se destruyan.
  • Bola reforzada: Un impacto (hasta 3 bloques).
  • Congelación: 8 segundos.
  • Plataforma expansiva: 15 segundos.
  • Atracción magnética: 15 segundos.
  • Ladrillo bomba: Un impacto.
  • Escudo inferior: 15 segundos o hasta un impacto de la bola.
  • Puntos dobles: 10 segundos.
  • Puntos triples: 10 segundos.
RF17.

El sistema debe generar aleatoriamente entre 0 y 3 cápsulas de power-up por nivel, escondidas entre los bloques del escenario.

RF18.

Al ser destruido el bloque que oculta una cápsula de power-up, esta se moverá en caída libre hacia el límite inferior de la pantalla; al atravesar el límite se entenderá por perdida.

RF19.

El sistema deberá activar un power-up al jugador solo cuando una cápsula de power-up en caída libre choque con la plataforma.

RF20.

El sistema debe mostrar la animación de destrucción de obstáculos cuando estos sean golpeados por un power-up, siempre que los efectos visuales estén activados.

RF21.

Se debe mostrar, en la esquina superior derecha de la pantalla de partida, el puntaje en tiempo real durante la partida.

RF22.

El sistema debe mostrar, en la esquina superior derecha de la pantalla de partida, a la izquierda del puntaje de la partida, el conteo en tiempo real de las vidas del usuario.

RF23.

El sistema debe permitir al usuario iniciar la partida con 5 vidas.

RF24.

El sistema debe restar una vida al usuario cuando la bola atraviesa el límite inferior de la pantalla de partida.

RF25.

El sistema debe iniciar la partida con el tiempo correspondiente al nivel actual.

RF26.

El sistema debe mostrar, en la esquina superior izquierda de la pantalla de partida, el conteo regresivo del tiempo.

RF27.

El sistema debe calcular el puntaje del nivel según la siguiente fórmula:


  • +10 puntos por cada bloque destruído.
  • +10 puntos por cada obstáculo no destruido.
  • -0.3 puntos por cada segundo transcurrido.
  • -5 puntos por cada vida perdida.
  • Total x1 si el nivel de dificultad es máximo, x0.8 si es moderado y x0.6 si es fácil.
RF28.

Al destruir todos los bloques, se debe avanzar al jugador al siguiente nivel.

RF29.

El sistema debe mostrar pantallas de victoria y derrota por pérdida de vidas o por agotamiento del tiempo.

RF30.

Al perder todas las vidas (vidas = 0), se entiende terminado el juego, se muestra la pantalla de derrota por vidas y se regresa al usuario a la pantalla de selección de nivel.

RF31.

Al acabarse el tiempo (tiempo = 0), se entiende terminado el juego, se muestra la pantalla de derrota por tiempo y se regresa al usuario a la pantalla de selección de nivel.

RF32.

El usuario puede consultar el récord de sus puntuaciones (por nivel y de manera general) de manera local.

RF33.

El usuario puede consultar y filtrar (por nombre y puntaje) el ranking global si hay conexión a internet.

RF34.

El sistema debe notificar si no hay conexión al intentar acceder al ranking.

RF35.

El sistema debe permitir seleccionar el nivel de dificultad (fácil, medio, difícil) antes de iniciar una partida.

RF36.

El sistema debe permitir personalizar los controles de movimiento de la plataforma.

RF37.

El sistema debe mostrar una pantalla de tutorial o ayuda que explique la mecánica básica del juego.

RF38.

El sistema debe mostrar una tabla de power-ups que incluya nombre, efecto, duración, condiciones de aparición y estado de desbloqueo.

RF39.

El sistema debe indicar visualmente qué power-ups han sido desbloqueados por el usuario y cuáles aún no.

RF40.

El sistema deberá activar un power-up al usuario cuando la cápsula de power up toque la plataforma.

RF41.

El sistema debe reproducir el efecto de sonido para el choque de la bola cuando esta:


  • Choca con la plataforma
  • Choca con las paredes.
  • Choca con un obstáculo.
RF42.

El sistema debe reproducir el efecto de sonido para la destrucción del bloque cuando un bloque sea destruido.

RF43.

El sistema debe reproducir el efecto de sonido para la destrucción del obstáculo cuando un obstáculo sea destruido.

RF44.

El sistema debe reproducir el efecto de sonido correspondiente a cada power-up.

RF45.

El sistema debe reproducir el efecto de sonido de pérdida de vida cuando la bola atraviesa el límite inferior de la pantalla de partida.

RF46.

El sistema debe reproducir las pistas de la banda sonora de forma automática, seleccionando la pista correspondiente según el tipo de pantalla (menú, partida, victoria, derrota). En la pantalla de partida reproducirá de manera aleatoria una de las 4 pistas disponibles para este tipo de pantalla.

RF47.

El sistema no deberá reproducir pista alguna en la pantalla de juego en pausa ni en la pantalla de tutorial.

RF48.

El sistema debe mostrar pantallas de transición entre niveles.

RF49.

El sistema debe permitir al usuario visualizar la información de cada nivel (nombre, número, dificultad) antes de jugarlo.

RF50.

El sistema debe permitir consultar desde el menú principal las puntuaciones personales, tanto generales como por nivel.

3.3 Requisitos no funcionales

RNF1.

Rendimiento: El juego debe ejecutarse a un mínimo de 30 FPS en equipos con procesador dual-core y 2GB de RAM.

RNF2.

Usabilidad: La interfaz debe ser intuitiva, permitiendo al usuario comenzar a jugar sin instrucciones complejas.

RNF3.

Compatibilidad: El juego debe poder ejecutarse en Windows, Linux y Mac.

RNF4.

Portabilidad: El sistema debe funcionar como archivo ejecutable único.

RNF5.

Seguridad: No se almacenará información personal sensible del usuario.

RNF6.

Mantenibilidad: El código debe estar modularizado y completamente documentado para facilitar su modificación.

RNF7.

Accesibilidad: Soporte visual para personas con visión reducida. Posibilidad de activar modo de alto contraste, modificación del tamaño de la fuente.

RNF8.

Fiabilidad: El sistema debe mantener un funcionamiento estable sin cuelgues ni pérdidas de datos durante sesiones de juego prolongadas.

RNF9.

Recuperación ante errores: ante errores no críticos, como pérdida de conexión con el servidor del ranking, el sistema debe mostrar un mensaje claro y permitir continuar el juego de forma local.

RNF10.

Instalación / Despliegue: El sistema debe poder instalarse o ejecutarse sin necesidad de configuración adicional por parte del usuario (plug-and-play).

RNF11.

Localización: El juego debe permitir soporte multilingüe, empezando con idioma inglés como predeterminado y posibilidad de expansión a otros idiomas.

RNF12.

Consumo de recursos: El sistema no debe utilizar más del 60% de CPU ni más de 500 MB de RAM durante el juego en condiciones normales.

RNF13.

Trazabilidad: Todos los requisitos funcionales deben estar trazados a funcionalidades específicas del sistema y cubiertos por pruebas.

RNF14.

Actualización: El sistema debe permitir actualizaciones manuales o automáticas sin requerir reinstalación completa del juego.

4. Conclusiones y reflexiones

El desarrollo de esta Especificación de Requisitos de Software ha evidenciado la importancia de establecer de forma clara, precisa y estructurada los requerimientos del videojuego Arkanoid Evolution (ArkEv), ya que ha permitido construir una visión integral del producto, facilitando la planificación y ejecución del proyecto. Mediante la adopción del modelo IEEE 830, se organizaron adecuadamente las funcionalidades del sistema, sus restricciones técnicas y atributos de calidad esperados. Esta estructura garantiza que todos los actores involucrados compartan una comprensión unificada del objetivo del desarrollo, favoreciendo así una mayor eficiencia y alineación en cada una de las etapas del ciclo de vida del software.
